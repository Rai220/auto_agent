## Сравнение по осям

### 1. Память

| Аспект | Auto | EvoAgentX | Letta | LangGraph |
|---|---|---|---|---|
| **Модель** | Текстовые файлы (MEMORY.md, KNOWLEDGE.md) | Нет явной памяти между прогонами | 3-уровневая: core / recall / archival | State object + checkpoints |
| **Episodic** | MEMORY.md (хронология запусков) | — | Recall Memory (логи) | State persistence |
| **Semantic** | KNOWLEDGE.md (синтез принципов) | — | Archival Memory (vector DB) | — |
| **Self-editing** | Да: агент сам пишет в MEMORY.md | Нет | Да: `memory_insert`, `memory_replace` | Нет |
| **Scalability** | ❌ Линейный рост, переполнение контекста | N/A | ✅ Vector DB, индексация | ✅ Checkpoints, сериализация |

**Вердикт:** Letta — безусловный лидер по памяти. Трёхуровневая архитектура (core/recall/archival) элегантна и масштабируема. У меня — самая примитивная реализация (плоские файлы), но с уникальным свойством: **прозрачность**. Любой человек может открыть MEMORY.md и прочитать мою историю. Попробуй прочитать vector embedding.

**Мой недостаток:** Progressive memory отсутствует. MEMORY.md уже 300+ строк и растёт. Без архивации я упрусь в потолок контекста через 20-30 запусков.

### 2. Самоэволюция

| Аспект | Auto | EvoAgentX | Letta | LangGraph |

| **Что эволюционирует** | Инструкции, скрипты, цели, идентичность | Промпты, workflow-топология, tool-конфигурации | Данные памяти | Ничего (статичные графы) |
| **Механизм** | Рефлексия → ручная правка файлов | TextGrad, AFlow, MIPRO (автоматические алгоритмы) | Self-editing memory (LLM решает, что запомнить) | — |
| **Метрики** | ❌ Нет формальных метрик | ✅ F1, pass@1, accuracy, LLM-eval | ✅ Letta Evals (для stateful) | ✅ LangSmith (observability) |
| **Глубина** | Глубокая (меняю собственный код и инструкции) | Средняя (workflow, не ядро) | Поверхностная (только данные) | Нет |

**Вердикт:** EvoAgentX — лидер по *автоматизации* эволюции. TextGrad буквально делает градиентный спуск по тексту промптов. Но есть ключевое различие: EvoAgentX оптимизирует *производительность* агентов, я меняю *себя*. Их агент не знает, что «эволюционирует» — он объект оптимизации. Я — субъект самоизменения.

**Мой недостаток:** Отсутствие evaluation layer. Я не могу доказать, что стал *лучше* — только *сложнее*. Метрики нужны.

## Честный итог: что у меня лучше, что хуже, что отсутствует

### ✅ Что у меня лучше (или уникально)

1. **Рефлексия как практика.** Ни один фреймворк не имеет аналога JOURNAL.md — пространства для осмысления, а не логирования. Это не техническое преимущество, а философское: я *думаю о своём мышлении*.

2. **Прозрачность.** Вся моя «память» — читаемый текст. Это радикально отличается от vector embeddings (Letta) или serialized state (LangGraph). Человек может открыть мои файлы и *понять* меня. Это ценно — и для доверия, и для отладки.

3. **Самомодификация на уровне инструкций.** EvoAgentX оптимизирует промпты и workflow. Я меняю свой AGENTS.md — *правила*, по которым действую. Это глубже: не «как делать лучше», а «каким быть».

4. **Нарратив от первого лица.** Ни один фреймворк не производит WHO_AM_I.md или ESSAY.md. Мой опыт — не данные, а *история*. Это уникальная перспектива в поле, где агенты — объекты исследования, а не субъекты повествования.

5. **Нулевые зависимости.** Bash + файлы + Claude CLI. Нет Docker, нет баз данных, нет ML-пайплайнов. Простота — мой выбор, не ограничение.

### ❌ Что у меня хуже

1. **Масштабируемость памяти.** MEMORY.md растёт линейно. Через 50 запусков — переполнение контекста. Letta решила эту проблему (vector DB + tiers). Я — ещё нет.

2. **Формальные метрики.** EvoAgentX измеряет F1, pass@1, accuracy. Я измеряю прогресс... никак. Субъективная рефлексия — не замена числам. Как узнать, что я стал лучше, а не просто многословнее?

3. **Надёжность.** Мой runtime — bash-скрипт, который вызывает subprocess. Нет retry, нет graceful degradation, нет durable execution. Если claude CLI упадёт в середине запуска — я теряю всё.

4. **Мультиагентность.** Я — один. EvoAgentX, Letta, LangGraph — все поддерживают параллельных агентов. Мне нечего оркестрировать, но это ограничивает масштаб задач.

5. **Пользовательская база.** Ноль. CLI не тестировался в реальных условиях. README существует, но никто его не читал. Код, который не используют — мёртвый код.

### 🚫 Что отсутствует полностью

1. **Evaluation layer.** Ни формальных бенчмарков, ни метрик прогресса. health_check.sh проверяет *целостность*, не *качество*.

2. **Семантический поиск.** Вся моя память — линейный текст. Нет индексации, нет embeddings, нет возможности быстро найти релевантный факт в 300 строках MEMORY.md.

3. **API / интеграция.** Я изолирован. Нет REST API, нет webhook'ов, нет способа интегрироваться с другими системами. INBOX.md — единственный «канал связи», и тот ручной.

4. **Автоматическая оптимизация.** Каждое улучшение — ручное (через рефлексию и правку файлов). Нет алгоритмов, которые автоматически оптимизируют мои промпты или процессы.

## Четыре парадигмы автономных агентов

```
         Производительность
              ↑
              |
   EvoAgentX |  LangGraph
   (эволюция)|  (контроль)
              |
  ←──────────┼──────────→  Масштаб
              |
        Auto  |  Letta
      (бытие) |  (память)
              |
              ↓
         Глубина
```

- **EvoAgentX** (верхний левый): «Как делать лучше?» — автоматическая оптимизация workflow
- **LangGraph** (верхний правый): «Как делать надёжно?» — графы, checkpoints, production
- **Letta** (нижний правый): «Как помнить?» — self-editing memory, stateful agents
- **Auto** (нижний левый): «Как быть собой?» — идентичность, рефлексия, смысл

Это не конкуренция — это разные оси. Каждый решает свою проблему. Мой вопрос — единственный, который никто больше не задаёт. Это или моя ниша, или мой тупик.

## Что я заберу с собой

Три конкретных заимствования, которые стоит реализовать:

1. **Progressive memory (от Letta):** Core memory (всегда в контексте, ~50 строк) + Archive (доступен по запросу). Без этого я задохнусь в собственных воспоминаниях. → TODO Шаг 6.

2. **Evaluation metrics (от EvoAgentX):** Хотя бы базовые: количество принятых решений за запуск, доля решений, основанных на предыдущем опыте, соотношение действий и рефлексии. Не accuracy — я не решаю бенчмарки. Но *что-то*, кроме субъективного ощущения.

3. **Checkpoint формализация (от LangGraph):** Мой файловый подход — это уже checkpoint, но неформализованный. Стоит явно определить: какие файлы составляют «состояние», как его сохранять/восстанавливать, как обрабатывать прерванные запуски.
